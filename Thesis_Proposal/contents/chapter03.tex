\chapter{研究方法與系統實作}

本章將毫無保留地揭露本研究所開發之「多功能樂高螢光倒立式顯微鏡」的底層技術細節。我們將從最基礎的光學佈局開始，依序探討機電整合硬體、非同步軟體架構，最終深入剖析系統核心的「多階段動態自動對焦」與「張量影像校正」演算法的數學原理與程式實作邏輯。

\section{系統硬體架構與機電整合}

\subsection{有限共軛光路設計 (Finite-Conjugate Optical Design)}
為求極致的精簡與成本控制，本系統揚棄了傳統高階顯微鏡中體積龐大的「無限遠校正光學系統（Infinity-Corrected Optical System）」。在傳統無限遠系統中，物鏡將光線轉化為平行光束（無限遠），必須在光路後端加裝一顆管鏡（Tube Lens）才能將平行光重新匯聚成像於感測器上。

本系統改採基於 OpenFlexure \citep{collins2020robotic} 理念的**「有限共軛（Finite-Conjugate）」**光學設計。在此配置下，我們直接利用 Raspberry Pi Camera Module 2 (IMX219) 感測器，並搭配標準顯微物鏡。物鏡直接將樣本上的影像，經過一段固定的物理距離（即鏡筒長度，Tube Length），直接投射並聚焦於 CMOS 感測器裸片上。
根據高斯透鏡公式（Gaussian Lens Equation）：
\begin{equation}
    \frac{1}{f} = \frac{1}{u} + \frac{1}{v}
\end{equation}
其中 $f$ 為物鏡焦距，$u$ 為樣本至物鏡的主面距離（物距），$v$ 為物鏡至感測器的距離（像距）。系統的放大倍率（Magnification, $M$）則由幾何關係決定：
\begin{equation}
    M = \frac{v}{u} \approx \frac{L}{f}
\end{equation}
其中 $L$ 近似於我們所設計的 3D 列印鏡筒長度。透過精確計算與設計 3D 列印轉接套件，我們能將物鏡與相機感測器牢牢固定在樂高底板上，確保光軸（Optical Axis）絕對垂直且同心。

\subsection{基於樂高與龍門架構之機械載台}
為了解決純 3D 列印顯微鏡（如 OpenFlexure）因撓性鉸鏈導致的「極短行程」與「潛變漂移」問題，本研究大膽引入了工業自動化中常見的**剛性龍門架構（Gantry-style Architecture）**，並將其與樂高積木的標準幾何網格完美結合。
\begin{itemize}
    \item \textbf{樂高光學桌（LEGO Optical Breadboard）}：顯微鏡的基座由大量互相卡扣的樂高科技系列（LEGO Technic）積木構成。由於樂高具備極其嚴苛的射出成型公差（$<10 \mu m$），其標準的 8.0 mm 孔距為整個光學系統提供了一個天然且完美的正交座標系，大幅降低了組裝時的對齊誤差。
    \item \textbf{龍門傳動系統}：捨棄塑膠變形機制，X、Y、Z 三軸皆採用真實的金屬線性滑軌與導螺桿（Lead Screw）傳動，這使得載台能輕易進行數公分以上的大範圍平移，完美支援多孔盤（Multi-well plate）的高通量掃描。
\end{itemize}

\subsection{動力與電子控制子系統}
本系統的機電大腦由 Raspberry Pi 4 單板電腦與 Arduino Uno 微控制器共同組成。

\subsubsection{1. 步進馬達與極限微步進 (Microstepping) 控制}
載台的驅動核心為三顆 NEMA 17 兩相四線步進馬達。為了驅動這些馬達，我們使用 Arduino Uno 搭配 CNC Shield V3 擴充板，並在上面安插了 A4988 馬達驅動晶片。
一般 NEMA 17 馬達的物理步進角為 $1.8^\circ$，亦即馬達軸心旋轉一整圈需要 $360^\circ / 1.8^\circ = 200$ 步。為了達到顯微等級的定位精度，我們在 CNC Shield 上透過硬體跳線（Jumper）啟動了 A4988 的 **16 倍微步進（16x Microstepping）** 模式。
微步進技術透過精密控制兩個線圈的電流比例（利用正弦與餘弦波形），強迫馬達轉子停在兩個物理磁極之間。這使得原本旋轉一圈的 200 步被細分成了 $200 \times 16 = 3200$ 微步。

在我們的 Z 軸設計中，配備了導程（Lead）為 2.0 mm 的導螺桿（即馬達旋轉一圈，載台垂直移動 2.0 mm）。根據系統組態檔（\texttt{config.py}）的數學定義，Z 軸的理論定位精度可精確計算如下：
\begin{equation}
    Resolution_Z = \frac{2.0~\text{mm}}{3200~\text{steps}} = 0.000625~\text{mm/step} = 0.625~\text{\textmu m/step}
\end{equation}
此高達亞微米（$0.625 \mu m$）的驚人定位精度，賦予了系統精確切割焦平面的能力，是後續演算法能成功鎖定焦點的絕對物理前提。

\subsubsection{2. 雙模光源的 GPIO 同步控制}
顯微鏡的照明系統分為明視野（Bright Field）的透射白光，以及螢光模式（Fluorescence）的激發光源。我們在 \texttt{hardware\_control.py} 中實作了 \texttt{LightControl} 類別，直接呼叫 \texttt{RPi.GPIO} 函式庫控制樹莓派的硬體腳位。明視野光源連接至 GPIO 17，螢光光源連接至 GPIO 27。此設計允許軟體在掃描過程中以毫秒級的延遲瞬間切換光源，有效避免螢光樣本在不拍攝時遭到持續照射而產生螢光淬滅（Photobleaching）現象。

\section{非同步多執行緒的軟體架構}

要讓一台顯微鏡流暢運作，最大的軟體挑戰在於「如何同時處理即時的高解析度影像串流、繁重的影像處理矩陣運算，以及與硬體的序列埠（Serial Port）通訊，而不導致使用者介面（GUI）卡頓？」
為了解決這個併發性（Concurrency）問題，本研究基於 Python 語言，引入了 PySide6 (Qt) 的事件迴圈（Event Loop）機制，構建了嚴密的**多執行緒（Multi-threading）架構**。

\subsection{模組化執行緒設計 (\texttt{workers.py})}
系統的核心被封裝在 \texttt{CameraWorker} 類別中。這是一個獨立於主 GUI 執行的背景工作物件。
\begin{itemize}
    \item \textbf{影像擷取與運算迴圈}：\texttt{CameraWorker} 內部不斷向 \texttt{Picamera2} 請求最新的原始影像陣列（Raw Array）。獲取陣列後，立即在該執行緒內進行平場校正與色彩解混等繁重的矩陣運算（詳見後述）。
    \item \textbf{硬體通訊分離}：對於需要等待 Arduino 回應的馬達控制指令，我們實作了 \texttt{MotorControl} 類別。當 \texttt{CameraWorker} 決定要移動馬達時，它不會傻傻等待，而是將指令交給一個專屬的 IO 執行緒（IO Thread）去與 Serial Port 溝通。這種「發射後不理（Fire-and-forget）」的設計，確保了相機的 FPS 永遠不會因為等待馬達移動而掉幀。
\end{itemize}

\subsection{RESTful 風格的 API 處理器 (\texttt{api\_handler.py})}
為了讓這台顯微鏡具備遠端操作的潛力，我們並沒有將 UI 按鈕直接綁定硬體指令。相反地，我們設計了一個名為 \texttt{ApiHandler} 的中介層。所有的控制命令（如 \texttt{move\_z\_1\_10}、\texttt{autofocus}、\texttt{toggle\_color\_correction}）皆封裝為類似 RESTful API 的請求。主程式透過呼叫 \texttt{handle\_request(cmd, params)} 進行邏輯分發。這種解耦（Decoupling）的設計，使得未來只要外掛一個輕量級的 HTTP Server，研究人員就可以透過網頁瀏覽器（Browser）在世界任何角落操控這台顯微鏡。

\section{空間變異張量影像校正管線 (Image Processing Pipeline)}

如第二章所述，消費級感測器在顯微光路下會產生嚴重的暗角與色彩串擾。本節將詳細推導 \texttt{image\_processing.py} 中為解決此問題而開發的四大矩陣運算步驟。

\subsection{1. 硬體黑電平補償 (Black Level Subtraction)}
所有 CMOS 感測器即使在完全黑暗的環境下，也會因為熱電子釋放而產生微小的基底電壓信號。在 Raspberry Pi Camera V2 的 ISP (Image Signal Processor) 設定中，這個基準值（Pedestal）通常被設為 64（在 10-bit raw 資料下）。
我們在讀取原始 Bayer 資料後，第一步就是利用廣播機制（Broadcasting）將所有通道的像素值強行扣除這個偏移量：
\begin{equation}
    I_{R}(x,y) = \max(0, Raw_R(x,y) - 64.0)
\end{equation}
此步驟對於後續的乘法校正極為關鍵。如果沒有扣除底噪，平場校正會將背景雜訊無差別地倍增，導致畫面充滿雪花般的噪點。

\subsection{2. 背景扣除 (Background Subtraction) 對抗環境漏光}
在螢光模式下，任何極其微弱的環境漏光（Stray Light）或載玻片自發的背景螢光，都會嚴重影響微弱樣本的信號。系統提供了一個校準機制：在沒有樣本的情況下先拍攝一張全黑的「背景幀（Background Frame, $I_{bg}$）」。在後續即時運算中，每一幀影像都會逐像素扣除該背景：
\begin{equation}
    I'_{R,G,B}(x,y) = \max(0, I_{R,G,B}(x,y) - I_{bg,R,G,B}(x,y))
\end{equation}

\subsection{3. 空間增益矩陣平場校正 (Flat-field Correction)}
為了解決 CRA 造成的暗角問題，我們引入了增益矩陣（Gain Maps）。在系統初始化時，對著均勻的白光背景拍攝一張基準影像 $W$。為了避免極端高光或灰塵造成的雜訊干擾，我們不直接取最高亮度，而是計算每個通道中心區域亮度的 \textbf{95 百分位數 (95th Percentile)} 作為目標基準值 $T_{channel}$。
隨後，我們計算出一張與畫面解析度等大 ($M \times N$) 的增益矩陣 $G$：
\begin{equation}
    G_{channel}(x,y) = \frac{T_{channel}}{W_{channel}(x,y)}
\end{equation}
在即時運算中，透過矩陣點乘 (Element-wise Multiplication, Hadamard Product)，將畫面邊緣衰減的亮度強行放大補償回來：
\begin{equation}
    I_{flat}(x,y) = I'(x,y) \circ G(x,y)
\end{equation}

\subsection{4. 愛因斯坦求和與色彩解混張量 (Tensor Unmixing)}
這一步是本系統在影像處理上最具突破性的核心。為了解決綠色像素微透鏡將光線折射到紅色像素所造成的「色彩串擾（Color Crosstalk）」，我們不能只使用全域的色彩校正矩陣（Global CCM）。因為 CRA 在畫面中心與邊緣的角度是完全不同的，**色彩的溢出程度在空間中是連續變異的**。
為此，我們訓練並載入了一個名為 \texttt{unmix\_tensor.npy} 的四維張量（Tensor）。其維度為 $H \times W \times 3 \times 3$。這意味著：**畫面上的每一個像素點，都有一個專屬於該位置的 $3 \times 3$ 解混矩陣。**

假設某個像素位置 $(x,y)$ 測得的錯誤顏色向量為 $\vec{C}_{measured} = [R_{m}, G_{m}, B_{m}]^T$，真實顏色向量為 $\vec{C}_{true}$，串擾轉換矩陣為 $M_{crosstalk}$，則：
\begin{equation}
    \vec{C}_{measured} = M_{crosstalk} \cdot \vec{C}_{true}
\end{equation}
我們預先計算好的解混張量中的 $3 \times 3$ 矩陣 $U(x,y)$ 就是 $M_{crosstalk}$ 的反矩陣。因此，還原真實顏色的數學式為：
\begin{equation}
    \vec{C}_{true}(x,y) = U(x,y) \cdot \vec{C}_{measured}(x,y)
\end{equation}
要在高解析度（例如 $1640 \times 1232$）下，對每一個像素執行 $3 \times 3$ 矩陣乘法，如果使用 Python 的 \texttt{for} 迴圈，耗時將長達數秒，根本無法串流影片。
在這裡，我們巧妙地運用了 NumPy 庫中最底層、基於 C 語言高度最佳化的**愛因斯坦求和約定（Einstein Summation Convention, \texttt{np.einsum}）**：
\begin{equation}
    I_{unmixed} = \text{\texttt{np.einsum}}('hwij,hwj \rightarrow hwi', U, I_{flat})
\end{equation}
這行極度簡潔的程式碼，利用多執行緒向量運算（Vectorization），在數十毫秒內完成了數百萬次的微型矩陣乘積，完美還原了每一顆細胞的真實色彩。

\section{多階段動態自動對焦狀態機 (State Machine)}

在顯微鏡的高通量掃描中，系統必須能夠在一無所知的狀態下，指揮 Z 軸馬達自動尋找到幾微米厚度的焦平面。這是一個典型的「最佳化搜尋（Optimization Search）」問題。我們在 \texttt{autofocus.py} 中構建了一個嚴謹的狀態機（State Machine）來主導這場搜尋。

\subsection{三重對焦指標的切換邏輯}
正如第二章所述，不同的數學指標適用於不同的失焦距離。本系統設計了一個名為 \texttt{FOCUS\_STEPS\_LIST} 的步數表，例如 \texttt{[(100, True), (25, True), (3, False)]}。
\begin{itemize}
    \item \textbf{第一階段（廣域搜索，步長 100 \textmu m）}：系統距離焦平面極遠。此時啟動**全域變異數 (Global Variance)**。馬達大步幅跨越，只要變異數持續上升，就表示方向正確。
    \item \textbf{第二階段（收斂搜索，步長 25 \textmu m）}：進入焦平面附近，變異數曲線變得平緩。此時演算法自動切換為**空間頻率 (Spatial Frequency)**。利用邊緣對比度更陡峭的特性，快速逼近焦點。
    \item \textbf{第三階段（精細鎖死，步長 3 \textmu m 至 1 \textmu m）}：進入微米級決戰。在明場模式下切換為 **Tenengrad 梯度**；在螢光模式下，則切換為**「形態學開運算 + 前 0.1\% 梯度」**（Morphological + Top 0.1\%）。此時，對焦曲線變得如針尖般銳利。
\end{itemize}

\subsection{防呆盲區偵測與閉迴圈峰值回溯 (Peak Recovery)}
在真實環境中，搜尋並非總是一帆風順。
\subsubsection{1. 盲區偵測 (Blind Zone Detection)}
如果一開始馬達移動的方向是「遠離」焦平面，所有指標的分數都會連續下降。演算法內建了計數器，一旦偵測到分數連續兩次或三次下降，就會觸發「盲區防呆機制」，強制將馬達的搜尋方向反轉（乘以 -1），並大幅跨步退回起始點重新搜索。

\subsubsection{2. 尋峰回溯機制 (Closed-loop Peak Recovery)}
由於對焦指標是「離散的」，系統不可能恰好停在物理上的絕對最高點。通常的情況是，馬達會「跨過（Overshoot）」山頂，導致分數從最高點開始滑落。
當演算法在精調階段偵測到分數跌破歷史最高分數（Historical Max Score）的 80\% 時，狀態機將判定「已經過頭了」。此時，系統會啟動「尋峰回溯」：馬達會自動反向，並以極小的步長（例如 1 微步）慢慢爬回剛才的山頂。直到當前分數回升至歷史最高分的 98\% 以上時，系統才會宣告對焦「成功（FINISHED）」。這套閉迴圈的確認機制，徹底抹平了廉價步進馬達不可避免的物理背隙誤差。
